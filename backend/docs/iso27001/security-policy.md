**Владелец документа:** CISO / Security Officer

## 1. Область применения

Настоящая политика определяет требования и меры по обеспечению информационной безопасности
CCTV Intelligence Platform v6.0 — платформы мониторинга камер видеонаблюдения, сбора телеметрии,
управления техническим обслуживанием (CMMS) и аналитики.

Политика охватывает:
- Все компоненты платформы (backend, frontend, мобильное приложение, P2P-шлюз)
- Данные, обрабатываемые, хранимые и передаваемые платформой
- Персонал, имеющий доступ к платформе
- Интеграции с внешними системами (CMMS, ServiceNow, Jira, 1С:ТОИР)

## 2. Цели безопасности (CIA Triad)

| Цель | Описание |
|------|----------|
| **Конфиденциальность** (Confidentiality) | Видеопотоки, телеметрия и данные ТО доступны только авторизованным пользователям |
| **Целостность** (Integrity) | Данные не могут быть изменены неавторизованно. HMAC-подпись аудита |
| **Доступность** (Availability) | Система доступна 24/7, целевой uptime 99.5% |

## 3. Классификация активов

### 3.1 Уровни классификации

| Уровень | Метка | Описание | Примеры |
|---------|-------|----------|---------|
| **L1** | `asset_class_critical` | Критические данные. Компрометация — остановка бизнеса | Видеопотоки, персональные данные, ключи шифрования |
| **L2** | `asset_class_confidential` | Конфиденциальные данные. Компрометация — репутационный ущерб | Данные ТО, логи аудита, конфигурация |
| **L3** | `asset_class_internal` | Внутренние данные. Компрометация — ограниченный ущерб | Телеметрия, метрики, отчёты |
| **L4** | `asset_class_public` | Публичные данные | Статусы камер (без координат), документация |

### 3.2 Классификация устройств

Каждому устройству присваивается `asset_class`:
- `critical` — камеры периметра, серверные NVR, критическая инфраструктура
- `confidential` — камеры внутренних помещений, точки входа
- `internal` — общие камеры, коммутаторы, датчики
- `public` — камеры публичных зон

## 4. Управление доступом (A.9)

### 4.1 Ролевая модель (RBAC)

| Роль | Права |
|------|-------|
| `admin` | Полный доступ: пользователи, устройства, конфигурация, CMMS |
| `owner` | Управление своими устройствами и техниками |
| `manager` | Просмотр всех устройств, управление ТО, отчёты |
| `technician` | Выполнение нарядов, просмотр назначенных устройств |
| `support` | Просмотр телеметрии и логов, создание тикетов |
| `viewer` | Только просмотр статусов устройств |

### 4.2 Аутентификация

- JWT access token (15 мин) + refresh token (24 часа)
- 2FA через TOTP (опционально, рекомендуется для admin/owner)
- Telegram 2FA (опционально)
- API Keys для внешних интеграций (с префиксным поиском)
- HMAC-подпись для webhook-уведомлений из CMMS

### 4.3 Принцип минимальных привилегий

- Пользователи получают только те права, которые необходимы для выполнения обязанностей
- API Keys имеют ограниченный срок действия и список разрешений
- Доступ к устройствам ограничен через owner_id и site_id

## 5. Криптография (A.10)

| Применение | Алгоритм | Параметры |
|------------|----------|-----------|
| Хеширование паролей | bcrypt | cost=12 |
| JWT подпись | HMAC-SHA256 | 256-bit secret |
| HMAC webhook | HMAC-SHA256 | per-system secret |
| HMAC аудит | HMAC-SHA256 | audit_hmac_key |
| Шифрование токенов (push) | AES-256-GCM | crypto.Encrypt/Decrypt |
| TLS | TLS 1.3 | минимальная версия |

## 6. Физическая безопасность (A.11)

- Серверы размещены в ЦОД с контролем доступа
- Видеорегистраторы и камеры в защищённых зонах
- Резервное питание (UPS)

## 7. Безопасность операций (A.12)

### 7.1 Защита от вредоносного ПО

- Еженедельное сканирование уязвимостей (Trivy)
- Автоматическое обновление зависимостей (Dependabot)
- Статический анализ кода (golangci-lint)

### 7.2 Резервное копирование

- Ежедневный бэкап PostgreSQL (pg_dump)
- Ретеншн TimescaleDB: телеметрия 30 дней, алармы 90 дней
- Плейбуки восстановления в [incident-response.md](incident-response.md)

### 7.3 Логирование и мониторинг

- Все действия пользователей логируются в `audit_log` (Приложение A.12.4)
- Webhook-уведомления логируются с HMAC-верификацией
- Конфликты синхронизации CMMS логируются с полным контекстом
- Аномалии детектируются Self-Healing Agent

### 7.4 Целостность журнала аудита

- Каждая запись `audit_log` подписывается HMAC-SHA256
- Ключ подписи хранится в `audit_hmac_key` (env/конфиг)
- Верификация целостности: API endpoint `GET /api/v1/audit/verify`
- При обнаружении несоответствия — алерт и блокировка

## 8. Безопасность коммуникаций (A.13)

### 8.1 Сетевая сегментация

- Backend API: порт 8080 (TLS)
- P2P Gateway: порт 8082 (TLS, API Key)
- GB28181 SIP: порт 5060 (UDP/TCP)
- SNMP Traps: порт 162 (UDP)
- Syslog: порт 1514 (UDP/TCP)

### 8.2 CORS Policy

- Белый список разрешённых origin (не `*`)
- Конфигурация: `cors_allowed_origins` в config.yaml
- По умолчанию: только домен фронтенда

## 9. Приобретение, разработка и поддержка (A.14)

- Весь код проходит code review
- Секреты не хранятся в коде (env-переменные / vault)
- CI/CD пайплайн включает проверку уязвимостей
- Зависимости регулярно обновляются

## 10. Отношения с поставщиками (A.15)

- Внешние CMMS-интеграции (ServiceNow, Jira, 1С:ТОИР) используют HMAC-подпись
- API Keys для внешних систем имеют минимальные права
- Регулярный аудит внешних интеграций

## 11. Управление инцидентами (A.16)

Подробный план реагирования: [incident-response.md](incident-response.md)

### Классификация инцидентов

| Severity | Описание | Время реакции |
|----------|----------|---------------|
| **P1 — Critical** | Утечка данных, компрометация системы | 15 минут |
| **P2 — High** | Отказ критического сервиса, подозрительная активность | 1 час |
| **P3 — Medium** | Несанкционированный доступ (попытка), сбой не-critical | 4 часа |
| **P4 — Low** | Minor policy violation, просроченный сертификат | 24 часа |
